# RESOLVER.md — 知識路由決策樹

> 每次要寫任何東西進 brain/，先走這個決策樹。一分鐘內找到正確目錄。

---

## 決策樹

```
這件事是關於什麼的？
│
├─ 某個系統/服務 反覆出錯、有已知修法、有歷史事件？
│   → bugs/{service-name}.md
│
├─ 「以後不能這樣做」的規則、禁令、教訓？
│   → rules/{domain}.md
│
├─ 技術選型、架構方向、「我們為什麼選 X 不選 Y」？
│   → decisions/{topic}.md
│
├─ 怎麼操作某個服務（啟動、停止、部署、排查指令）？
│   → ops/{service-name}.md
│
└─ 不確定放哪？
    → inbox/YYYY-MM-DD-{topic}.md  ← inbox 上限 10 條，滿了就處理
```

---

## 各目錄定義

### bugs/
**放什麼**：已知會反覆出問題的系統/服務，含已知根因、修法、未解問題。
**不放什麼**：純操作手冊（→ ops/）、技術選型（→ decisions/）
**格式**：事實摘要（動態更新）+ 事件時間線（只 append）

### rules/
**放什麼**：從失敗中蒸餾出來的「永遠不能這樣做」規則。
**不放什麼**：某次事件的記錄（→ bugs/）、操作流程（→ ops/）
**格式**：規則清單 + 來源（error-log / law.json）

### decisions/
**放什麼**：技術選型決策、架構方向，含「為何棄用 X」。
**不放什麼**：操作問題（→ bugs/）、強制規則（→ rules/）
**格式**：決策 + 理由 + 替代方案 + 日期

### ops/
**放什麼**：服務操作手冊：啟動、停止、重啟、部署指令、日誌路徑。
**不放什麼**：錯誤歷史（→ bugs/）、規則（→ rules/）
**格式**：指令優先，說明次之

### inbox/
**放什麼**：不確定分類的暫存項目。
**規則**：
- 上限 10 條
- 超過 → 強制處理（分類或刪除），不能繼續塞
- 每週一清空
- 每條都要有日期，7 天未處理自動歸檔

---

## 黃金規則

1. **一個實體，一個文件**。不要每天建新文件，而是更新同一個。
2. **頂部 = 現在知道的最佳事實**（隨時更新）。
3. **底部 = 事件時間線**（只能 append，不能修改）。
4. **law.json 是機器讀的規則**，rules/ 是人讀的規則，兩者互補不重複。
5. **不知道放哪 = inbox**，但 inbox 不是垃圾桶。
