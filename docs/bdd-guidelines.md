
# BDD（行為驅動開發）架構級規範

## 一、核心原則
- 所有需求、流程、bug 修復，必須先以「人話」行為描述（Given-When-Then）明確定義預期行為與驗證標準。
- BDD 不僅驗證功能，也驗證流程正確性、權限、資料流、異常處理、追溯鏈與用戶體驗。
- BDD 案例必須可自動化執行，並與需求、設計、知識文件雙向鏈結。

## 二、流程要求
1. **需求澄清**：每個新需求、bug、流程優化，必須先寫 Given-When-Then 行為描述，經需求方/技術方雙方確認。
2. **案例落地**：每個 BDD 案例必須對應一組自動化測試（如 pytest-bdd、behave、Gherkin）。
3. **全流程覆蓋**：高風險/核心流程必須有完整 BDD 覆蓋，需求變更時同步更新。
4. **驗證與追溯**：每個案例需標註對應需求、知識文件、驗證報告 ID，方便追溯。
5. **CI 強制**：BDD 測試必須納入 CI pipeline，未通過禁止合併。

## 三、技術棧適用細則
- **Python/核心邏輯**：推薦 pytest-bdd、behave，所有核心模組須有 Given-When-Then 覆蓋。
- **API 層**：每個 API endpoint 必須有 BDD 案例描述權限、資料驗證、異常、用戶行為。
- **資料層/狀態同步**：資料一致性、同步、rollback、異常流程都需有 BDD 案例。
- **RAG/知識庫**：知識入庫、查詢、去重、追溯等流程必須有 BDD 行為描述與自動化驗證。
- **驗證/閉環層**：自動恢復、CAPA、closeout 等閉環流程必須有 BDD 案例覆蓋「觸發-驗證-沉澱」全流程。

## 四、範例（Gherkin 語法）
```gherkin
Feature: Transactional status update
  Scenario: 成功提交原子狀態
    Given 系統初始狀態為 A
    When 用戶提交狀態更新請求
    Then 狀態應正確更新且資料一致

  Scenario: 狀態更新失敗時自動 rollback
    Given 系統初始狀態為 A
    When 用戶提交異常請求
    Then 狀態應維持原樣且產生異常紀錄
```

## 五、驗證標準
- 覆蓋率：所有高風險/核心流程 100% 覆蓋，其他流程 80% 以上。
- 報告：CI pipeline 產生 BDD 測試報告，並自動存檔。
- 追溯：每個案例可追溯到需求、知識文件、驗證報告。
- 失敗閉環：每次流程失敗必須有對應 BDD 案例驗證修復與知識沉澱。

## 六、常見反模式（嚴禁）
- 只寫 happy path，未覆蓋異常/權限/資料一致性。
- BDD 案例與需求、知識文件脫鉤，無法追溯。
- 案例僅手動驗證，未自動化。
- 需求變更未同步更新 BDD 案例。
- 測試未納入 CI pipeline 或未強制通過。
